# Chapter 7: Conclusion and Further Work

## 7.1 Project Summary

This project set out to develop a multi-vendor fashion e-commerce platform that supports the full marketplace workflow for both customers and vendors. The scope was defined by three compulsory specification blocks — Block A (Core Functions), Block B (Multiple Photos and Order Processing), and Block F (Multiple Vendors) — together with three optional blocks — Block T (User Generated Content), Block U (Wishlist and Promotional Pricing Strategy), and Block X (User Experience Design). The system was implemented as a React single-page application connected to a Django REST Framework backend with an SQLite database.

The customer-facing storefront delivers product browsing and search without authentication, a persistent server-side shopping cart, multi-vendor split checkout, order history with status filtering, and a product-detail experience that includes multi-image galleries and verified-purchase reviews. The vendor-facing management workspace provides shop creation and profile maintenance, product catalogue operations including creation, editing, visibility toggling, and multi-image upload, a four-state order-processing workflow covering Pending, Holding, Shipped, and Cancelled states, refund-request handling, promotional pricing with configurable discount periods, and an analytics dashboard driven by recorded product views, search queries, wishlist data, and sales figures.

Engagement features connect the two sides of the marketplace. Wishlist sale notifications inform customers when a saved product enters a promotion, event-driven in-app notifications cover order status changes and refund outcomes, and the analytics dashboard helps vendors identify high-interest products and recommend discount candidates. The interface adapts to desktop and mobile screen sizes through responsive layouts, a mobile drawer menu, swipe-enabled image galleries, and persistent dark-mode support.

## 7.2 Requirements Satisfaction

Block A (Core Functions) required twenty storefront and administration features. All twenty have been implemented. Customer functions A1 through A13 — covering registration, product browsing and search, cart management, checkout, and order history — are fully operational. Vendor administration functions A14 through A20 — covering catalogue maintenance, product visibility control, and order processing — are likewise complete.

Block B (Multiple Photos and Order Processing) specified four requirements, of which three are fully satisfied. Products support multiple images displayed in a gallery on the storefront and managed through the vendor portal (B1). The order lifecycle follows a four-state workflow in which both customers and vendors trigger status changes (B2), and customers can filter their order list by status (B3). The remaining requirement, B4, asked for per-transition date recording. The system partially satisfies this by recording a general last-updated timestamp whenever the status changes, but it does not maintain separate dedicated dates for each individual transition.

Block F (Multiple Vendors) defined five requirements and all five are met. Vendors register and operate independent shops with their own catalogues and order queues (F1, F2). Customers can visit a shop page, view its profile, and search within that shop's products (F3). The cart accepts products from multiple shops, and checkout automatically splits the selected items into one purchase order per vendor (F4). Items not selected for checkout remain in the cart for future purchase (F5). F1 is adapted slightly in that shop photos are uploaded through the store management interface after registration rather than on the initial registration form.

Block T (User Generated Content) is fully implemented. Customers can rate purchased products on a five-point scale, write text reviews, and attach image media. The system enforces one review per customer per product and restricts submission to verified purchasers who have received the item in a shipped order, ensuring that all published reviews originate from authentic buyers.

Block U (Wishlist and Promotional Pricing Strategy) is fully implemented. Customers can add products to a personal wishlist, and vendors can create time-limited promotional discounts. When a promotion becomes active for a wishlisted product, the system automatically generates a sale notification for the affected customers. Product views and search queries are logged and surfaced through the vendor analytics dashboard alongside wishlist popularity rankings and data-driven discount recommendations.

Block X (User Experience Design) is fully implemented. The interface provides responsive layouts for desktop and mobile devices, a mobile navigation drawer, swipe-enabled product image galleries, persistent dark-mode support, pagination controls for large catalogues, notification badges with a notification panel, and feedback messages for user actions.

In summary, all three compulsory blocks are satisfied, with only B4 partially met, and all three optional blocks are fully implemented. The delivered system therefore meets the requirements identified at the start of the project.

## 7.3 Limitations

Despite satisfying the chosen specification blocks, the project has several limitations that would need to be addressed before the system could operate in a production environment.

The application uses SQLite as its sole data store. SQLite is file-based and does not support concurrent write access efficiently, which restricts the system to single-user or low-traffic scenarios. A production deployment would require migration to a client–server database engine such as PostgreSQL. Closely related is the absence of any external payment gateway integration; orders are created and tracked, but no monetary transaction is processed, and checkout assumes payment has been completed without verification.

User authentication relies on a custom device-token scheme without email verification on registration, password-reset functionality, or third-party OAuth login through providers such as Google or Facebook. These omissions reduce both account security and user convenience. Product search is implemented through Django ORM substring matching, with no full-text search engine, relevance ranking, stemming, or fuzzy matching, which would cause performance and result-quality degradation as the product catalogue grows.

As noted under requirement B4, the order model records a single last-updated timestamp rather than separate dates for each status transition, meaning that earlier transition dates are overwritten by later ones. The notification system is limited to in-app delivery; the system does not send email notifications, push notifications, or SMS alerts, which reduces its ability to reach users who are not actively browsing the application.

Uploaded product images are stored on the local filesystem at their original resolution without compression, resizing, or thumbnail generation, and there is no content delivery network integration or conversion to optimised formats such as WebP. Although the data model defines a video media type for products, the upload field uses an image-specific field and the frontend does not include a video player component, so video attachments are not fully functional in practice.

The backend test file is empty, providing no automated unit, integration, or API test coverage. The application is available in English only; although Django's internationalisation setting is enabled, no translation files or frontend localisation libraries have been added. Finally, system-level administration relies entirely on Django's built-in admin interface, with no custom admin dashboard offering platform-wide analytics, user management, or content moderation workflows.

## 7.4 Further Work

The limitations described above suggest several directions for future development that would move the system from a project-scope prototype toward a production-ready marketplace.

The most impactful infrastructure change would be migrating the database from SQLite to PostgreSQL. This would improve write concurrency, enable multi-instance deployment behind a load balancer, and open the possibility of using PostgreSQL's built-in full-text search capabilities to replace the current substring matching with relevance-ranked results, stemming, and fuzzy matching. Alternatively, a dedicated search engine such as Elasticsearch could be integrated for more advanced catalogue search requirements.

Connecting the checkout flow to a payment provider such as Stripe or PayPal would allow the system to process real transactions. This would involve creating a payment intent before order confirmation, handling success and failure callbacks, and recording payment status alongside order status. On the authentication side, adding email verification at registration, a password-reset flow via email tokens, and OAuth sign-in through external identity providers would improve both security and the registration experience, while API rate limiting through Django REST Framework's throttling classes would protect authentication endpoints from brute-force attacks.

Extending the notification system to include email delivery and mobile push notifications via Firebase Cloud Messaging would ensure that customers and vendors receive timely updates even when they are not actively using the application. Introducing server-side image resizing and thumbnail generation through a library such as django-imagekit, together with WebP conversion and CDN integration, would reduce page-load times and bandwidth consumption. Making the video media type functional would require replacing the image-specific upload field with a general file field for video entries, adding a frontend video player component, and validating video formats and file-size limits on the server.

Extending the order model to record a separate timestamp for each status transition — for example through a related OrderStatusHistory model that logs every state change with its date — would fully satisfy requirement B4 and give both customers and vendors a complete audit trail of order progress. Building an automated test suite covering model validation, serializer logic, API endpoint behaviour, and permission checks would provide regression safety for future development, and frontend component tests using React Testing Library would complement the backend coverage.

Adding Django translation files and a frontend localisation library such as react-i18next would allow the interface to be offered in multiple languages, broadening the potential user base. Implementing real-time chat between customers and vendors through Django Channels with WebSocket support would enable pre-sale enquiries and post-sale communication without leaving the platform. Integrating with courier APIs to surface real-time shipment tracking within the order-detail view would give customers visibility into the delivery progress of dispatched orders, which is a standard expectation of modern e-commerce systems. Together, these extensions would address the current limitations and bring the system closer to a commercially viable multi-vendor marketplace.
