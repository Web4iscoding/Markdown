# Chapter 7: Conclusion and Further Work

## 7.1 Summary

This project developed a multi-vendor fashion e-commerce platform implemented as a React single-page application connected to a Django REST Framework backend with an SQLite database. The platform serves two primary user roles — customers who browse, purchase, and review products across multiple vendor shops, and vendors who manage their own catalogues, process orders, run promotions, and monitor sales analytics.

The project followed a structured development process. Requirements were analysed in Chapter 3 by studying established e-commerce platforms, mapping common marketplace patterns to the needs of a fashion-oriented multi-vendor system, and selecting six specification blocks that together define the scope of the project. Three compulsory blocks — Block A (Core Functions), Block B (Multiple Photos and Order Processing), and Block F (Multiple Vendors) — established the baseline, while three optional blocks — Block T (User Generated Content), Block U (Wishlist and Promotional Pricing Strategy), and Block X (User Experience Design) — extended the system with engagement and usability features. The system design documented in Chapter 4 translated these requirements into an architectural plan covering the three-tier structure, data modelling, and dynamic workflows. Chapter 5 then presented the project outcome by describing the delivered features and their supporting interfaces.

All twenty core functions specified by Block A have been implemented. Customers can register, browse and search products before or after login, manage a persistent server-side cart, check out with automatic order splitting by vendor, and review their order history with status filtering. Vendors can register a shop, maintain a product catalogue with multi-image support, process orders through a four-state lifecycle, and handle refund requests. Of Block B's four requirements, three are fully satisfied; B1 (multiple product photos with gallery display and vendor management), B2 (four-state order workflow with customer and vendor actions), and B3 (order status filtering) are complete. B4 (per-transition date recording) is partially met because the system records a general last-updated timestamp rather than separate dates for each individual status change. All five requirements of Block F are satisfied, including independent vendor shops, per-shop product catalogues, customer-facing shop pages with in-shop search, automatic cart splitting at checkout, and retention of unselected cart items.

The three optional blocks are fully implemented. Block T provides verified-purchase reviews with five-point ratings, text comments, and optional image attachments, enforcing one review per customer per product and restricting submission to buyers who have received the item in a shipped order. Block U delivers wishlist management, vendor-created time-limited promotions, automated sale notifications for customers who have wishlisted a promoted product, and a vendor analytics dashboard that surfaces search queries, product views, wishlist popularity, and data-driven discount recommendations. Block X is addressed through responsive layouts for desktop and mobile, a mobile navigation drawer, swipe-enabled image galleries, persistent dark-mode support, pagination controls, notification badges, and feedback messages for user actions.

The correctness of the implementation was verified through manual testing against each specification requirement. Every item in Blocks A, B, F, T, U, and X was exercised through the application interface to confirm that the expected behaviour matched the specification. The per-requirement implementation status is documented in the tables presented in Chapter 3, Section 3.3, which record full, partial, or adapted outcomes for each entry. The only requirement not fully satisfied is B4, as described above. All remaining requirements across all six blocks are met, and the delivered system therefore satisfies the scope defined at the start of the project.

## 7.2 Limitations

Despite satisfying the chosen specification blocks, the project has several limitations that would need to be addressed before the system could operate in a production environment.

The application uses SQLite as its sole data store. SQLite is file-based and does not support concurrent write access efficiently, which restricts the system to single-user or low-traffic scenarios. A production deployment would require migration to a client–server database engine such as PostgreSQL. Closely related is the absence of any external payment gateway integration; orders are created and tracked, but no monetary transaction is processed, and checkout assumes payment has been completed without verification.

User authentication relies on a custom device-token scheme without email verification on registration, password-reset functionality, or third-party OAuth login through providers such as Google or Facebook. These omissions reduce both account security and user convenience. Product search is implemented through Django ORM substring matching, with no full-text search engine, relevance ranking, stemming, or fuzzy matching, which would cause performance and result-quality degradation as the product catalogue grows.

As noted under requirement B4, the order model records a single last-updated timestamp rather than separate dates for each status transition, meaning that earlier transition dates are overwritten by later ones. The notification system is limited to in-app delivery; the system does not send email notifications, push notifications, or SMS alerts, which reduces its ability to reach users who are not actively browsing the application.

Uploaded product images are stored on the local filesystem at their original resolution without compression, resizing, or thumbnail generation, and there is no content delivery network integration or conversion to optimised formats such as WebP. Although the data model defines a video media type for products, the upload field uses an image-specific field and the frontend does not include a video player component, so video attachments are not fully functional in practice.

The application is available in English only; although Django's internationalisation setting is enabled, no translation files or frontend localisation libraries have been added. System-level administration relies entirely on Django's built-in admin interface, with no custom admin dashboard offering platform-wide analytics, user management, or content moderation workflows.

## 7.3 Future Work

The limitations described above suggest several directions for future development that would move the system from a project-scope prototype toward a production-ready marketplace.

The most impactful infrastructure change would be migrating the database from SQLite to PostgreSQL. This would improve write concurrency, enable multi-instance deployment behind a load balancer, and open the possibility of using PostgreSQL's built-in full-text search capabilities to replace the current substring matching with relevance-ranked results, stemming, and fuzzy matching. Alternatively, a dedicated search engine such as Elasticsearch could be integrated for more advanced catalogue search requirements.

Connecting the checkout flow to a payment provider such as Stripe or PayPal would allow the system to process real transactions. This would involve creating a payment intent before order confirmation, handling success and failure callbacks, and recording payment status alongside order status. On the authentication side, adding email verification at registration, a password-reset flow via email tokens, and OAuth sign-in through external identity providers would improve both security and the registration experience, while API rate limiting through Django REST Framework's throttling classes would protect authentication endpoints from brute-force attacks.

Extending the notification system to include email delivery and mobile push notifications via Firebase Cloud Messaging would ensure that customers and vendors receive timely updates even when they are not actively using the application. Introducing server-side image resizing and thumbnail generation through a library such as django-imagekit, together with WebP conversion and CDN integration, would reduce page-load times and bandwidth consumption. Making the video media type functional would require replacing the image-specific upload field with a general file field for video entries, adding a frontend video player component, and validating video formats and file-size limits on the server.

Extending the order model to record a separate timestamp for each status transition — for example through a related OrderStatusHistory model that logs every state change with its date — would fully satisfy requirement B4 and give both customers and vendors a complete audit trail of order progress. Building an automated test suite covering model validation, serializer logic, API endpoint behaviour, and permission checks would provide regression safety for future development, and frontend component tests using React Testing Library would complement the backend coverage.

Adding Django translation files and a frontend localisation library such as react-i18next would allow the interface to be offered in multiple languages, broadening the potential user base. Implementing real-time chat between customers and vendors through Django Channels with WebSocket support would enable pre-sale enquiries and post-sale communication without leaving the platform. Integrating with courier APIs to surface real-time shipment tracking within the order-detail view would give customers visibility into the delivery progress of dispatched orders, which is a standard expectation of modern e-commerce systems. Together, these extensions would address the current limitations and bring the system closer to a commercially viable multi-vendor marketplace.
